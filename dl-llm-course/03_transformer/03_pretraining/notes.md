# L3.3 预训练范式 · 笔记

## 一、两步范式:预训练 + 微调

```text
旧时代(从零训):
  随机权重 --[标注的任务数据]--> 任务模型
  每个任务都从随机权重开始,都要大量标注

新时代(预训练+微调):
  随机权重 --[海量无标注文本,自监督]--> 预训练模型(有通用语言能力)
                                              ↓
                                    --[少量任务标注数据]--> 任务模型(微调)
```

### 预训练 vs 从零训的核心区别

|        | 从零训一个任务            | 预训练                  |
| ------ | ------------------------- | ----------------------- |
| 起点   | 随机权重                  | 随机权重                |
| 数据   | **有标注**(影评+情感标签) | **无标注**(原始网页/书) |
| 监督   | 人工标签                  | 自监督(文本自己)        |
| 目的   | 学**这个任务**            | 学**通用语言能力**      |
| 数据量 | 小(标注贵)                | 海量(网页无限)          |

**两者都从随机权重开始，都"用现成内容"**。区别在:

- 从零训:内容是**为某个任务标注的**，学的是**任务专用**
- 预训练:内容是**无标注原始文本**，学的是**通用语言基础**

### 关键概念:自监督(Self-supervised)

预训练的监督信号来自**文本本身**--预测下一个词时，下一个词就是天然答案，不用人标注。这就是为什么能用海量网页/书籍(它们没标注，但能用来"预测下一个词")。

### 为什么预训练+微调更好

预训练模型已经懂语言了，微调时用少量标注就能适配任务--因为不用再从"学语言"开始，只需学"任务特性"。

**类比**:

- 从零训:招个文盲，从识字开始培训--要海量培训
- 预训练+微调:招个中文系毕业生(已懂语言)，培训两天就上手--少量培训

## 二、两种预训练目标:BERT vs GPT

### 路线 A:GPT 式--自回归(Autoregressive)

**任务**:给定前文，预测下一个词。

```text
输入: "今天 天气 真"
目标: 预测 "好"
```

L3.2 写的字符级 LM 就是 GPT 式(只是用 token 不是字符)。

- **方向**:从左到右，**单向**(只能看左边，靠 mask)
- **架构**:Decoder-Only
- **擅长**:生成(写文章、对话、代码)
- **代表**:GPT 系列、Claude、Llama、Qwen

### 路线 B:BERT 式--掩码语言模型(MLM)

**任务**:随机遮住一些词，让模型猜被遮的词。

```text
原文: "今天 天气 真 好"
掩码: "今天 天气 [MASK] 好"    ← 把"真"遮住
目标: 预测 "[MASK]" 位置是 "真"
```

- **方向**:**双向**--遮住的词左右两边都能看(无 mask，encoder 结构)
- **架构**:Encoder-Only
- **擅长**:理解(分类、问答、NER)
- **代表**:BERT、RoBERTa、DeBERTa

### 核心对比

|          | BERT(MLM)       | GPT(自回归)      |
| -------- | --------------- | ---------------- |
| 任务     | 猜被遮的词      | 预测下一个词     |
| 方向     | 双向(左右都看)  | 单向(只看左)     |
| 架构     | Encoder-Only    | Decoder-Only     |
| mask     | 无(全看)        | 有(挡未来)       |
| 擅长     | 理解            | 生成             |
| 典型任务 | 分类、问答、NER | 写作、对话、补全 |
| 代表     | BERT            | GPT/Claude/Llama |

**关键 insight**:BERT 和 GPT 的根本区别不在架构细节，而在**预训练目标**。同样的 Transformer，训练目标不同，能力方向完全不同。

**为什么现在 GPT 赢了**:生成任务(对话、写作)更通用、更有商业价值。大模型时代"生成即理解"--GPT 式成为主流。

## 三、tokenization:文本怎么变成数字

### 三种 tokenization

**1. 字符级**(L3.2 用的)

```text
"hello" -> ['h', 'e', 'l', 'l', 'o']    5 个 token
```

- 词表小(英文 ~100，中文 ~几千)
- 序列长
- 模型要自己学字符组成词，难

**2. 词级**

```text
"hello world" -> ['hello', 'world']    2 个 token
```

- 词表巨大(几十万词)
- 罕见词、新词、拼写错误没法处理(变 unknown)

**3. 子词级 BPE(实际用的)**

```text
"unhappiness" -> ['un', 'happ', 'iness']
"playing" -> ['play', 'ing']
```

**BPE(Byte Pair Encoding)**:把文本切成字符，然后**高频字符对合并**，迭代直到达到目标词表大小。

**为什么 BPE 是主流**:

- 词表适中(几万，GPT-2 用 50257)
- 能处理新词、罕见词(拆成已知子词)
- 常见词是整体(如 "the")，罕见词拆开
- 序列长度适中

**代表**:GPT 用 BPE，Llama 用 SentencePiece(BPE 变体)。

**token 数 ≠ 字符数 ≠ 词数**--GPT API 按 token 计费的原因。

## 四、Scaling Law(规模法则)

### 现象:模型越大、数据越多、算力越多，效果越好

OpenAI 2020 发现模型 loss 遵循可预测的幂律:

$$
\text{loss} \approx A / N^{\alpha} + B / D^{\beta} + C
$$

- N:参数量，D:数据量
- 给更多参数 + 数据 + 算力，loss 按**可预测曲线**下降，没有天花板

### Chinchilla 定律(2022,DeepMind)

GPT-3 有 1750 亿参数，但只训了 3000 亿 token--**数据不够**。Chinchilla 证明:**最优配比是参数量 ≈ 数据量(tokens)**。

- GPT-3:175B 参数 / 300B tokens ≈ 1:1.7(数据少了)
- 最优:175B 参数应该配 ~3.5T tokens

**这就是为什么 Llama 等用更少参数 + 更多数据反而更好**--配比对了。

### 三要素权衡

| 要素       | 代价          | 效果     |
| ---------- | ------------- | -------- |
| 参数量 N   | 显存、推理慢  | 容量大   |
| 数据量 D   | 标注/收集成本 | 学得充分 |
| 算力 FLOPs | GPU 钱、时间  | 训练充分 |

Chinchilla 给出最优配比:固定算力预算下，如何分配 N 和 D 让 loss 最小。这是大模型公司的核心竞争力。

### 实验:用 HuggingFace 体验 GPT-2 和 BERT

- 代码:[lab/play_pretrained.py](lab/play_pretrained.py)
- GPT-2 生成流畅英文(vs L3.2 玩具 GPT 的重复退化)

### 关键观察:预训练模型会"学坏"

BERT 预测出 hell/fuck/heck--**在统计上是对的**(网上这些词高频),但暴露了问题:**模型从数据里学到了粗俗/有毒表达**。

- 这正是 **Stage 6 安全专题**的核心议题:预训练模型从海量网页学语言,也学到偏见、毒性、有害内容
- **对齐(Stage 5)和安全(Stage 6)就是来解决"模型会说不该说的话"**
- BERT 没经过对齐(纯预训练模型),所以直接吐粗话;Claude/GPT-4 经过 RLHF 对齐,就不会

### 对比总结(实验结论)

```text
L3.2 玩具 GPT: 50 万参数 + 600 字符 -> 重复退化文本
GPT-2:         1.24 亿参数 + 40GB 预训练 -> 流畅英文
BERT:          1.1 亿参数 + 16GB 预训练 -> 遮词预测准确(但会"学坏")
```

**架构一样,数据和规模决定能力**。这就是 L3.3 的核心结论,也是大模型时代的基石。

## 八、补充问答

### `tokenizer.decode([tid])` 是什么

decode 是 encode 的逆操作--把 token id 序列转回文本。`encode` 把文本转成 token 在词表中的 id,`decode` 把 id 转回文本。传 `[tid]`(列表)因为 decode 接受 id 序列,不接单个 int。

### `nonzero(as_tuple=True)` vs `nonzero()`

- `nonzero()`:返回 `(n, ndim)` 2D 张量,每行一个完整下标
- `nonzero(as_tuple=True)`:返回元组,每维一个 `(n,)` 1D 张量,方便直接索引

1D 输入:默认 `(2,1)` 2D;as_tuple=True 是 `(tensor,)` 元组,`[0]` 取出干净的 1D 张量。BERT 代码里要 1D 下标做 `logits[0, mask_idx, :]`,所以用 as_tuple=True。
